### Standard File Order (Progressive)

| Position | File | Purpose | User Journey |
|----------|------|---------|--------------|
| 0 | overview.mdx | What & Why | "What is this package?" |
| 1 | get-started.mdx | Quick setup | "How do I set up?" |
| 2 | architecture.mdx | How it works | "How is it structured?" |

| 4 | `[specialized]`.mdx | Deep dive | "What about advanced topics?" |
| 5 | `[build, integration]`.mdx | Build & integration | "How are artifacts generated?" |
| 6 | `[testing, quality, checking]`.mdx | Testing & quality | "How do I test?" |
| 7 | `[release, publish, deploy]`.mdx | Release & deployment | "How do I release?" |
| 8 | troubleshooting.mdx | Debug | "Something's wrong" |

:::note Flexibility
This is a **guideline**, not strict rules. Adapt based on package needs:

- **Skip files** â€” Not every package needs `architecture.mdx` or `troubleshooting.mdx`
- **Merge files** â€” Combine related topics (e.g., `quality-check-build-deploy.mdx`)
- **Separate files** - For more complex explanation you can have both `testing.mdx` and `quality.mdx`, etc.
- **Add files** â€” Insert specialized docs where they make sense
- **Reorder** â€” Adjust positions if it improves the user journey

The goal is a **logical, progressive flow** â€” not rigid conformity.
:::

## Common Page Types

### get-started.mdx

| Section | Content |
|---------|---------|
| Prerequisites | Link to shared setup |
| Package Location | Filesystem path to the package |
| Development Commands | Dev server, watch mode (no lint/test) |
| Next Steps | Link to architecture, workflow, testing |

:::note
Lint, typecheck, and test commands belong in **testing.mdx** (position 7), not get-started.
:::

### architecture.mdx

| Section | Content |
|---------|---------|
| Project Structure | Directory tree with descriptions |
| Key Files | Important files explained |
| Data Flow | How data moves through the system |
| Design Decisions | Why things are the way they are |
| Next Steps | Link to core workflow |

### publishing.mdx

| Section | Content |
|---------|---------|
| Before Opening a PR | Checklist: build, local checks, update docs, create changeset |
| CI Pipeline | On Every PR (mermaid + table), On Merge to Main (mermaid), workflow file links |
| Package Distribution | For npm packages: exports table, published files |
| Deployment | For non-npm (e.g., docs site): deployment targets, features |
| Next Steps | Links to changeset workflow, testing, troubleshooting |

:::note
- **npm packages** use "Package Distribution" (exports, published files)
- **Non-npm packages** (like docs site) use "Deployment" (GitHub Pages, etc.)
- Always include mermaid diagrams for CI pipelines
- Always link to workflow files with ğŸ“„ prefix
:::
